MySQL数据类型

发表于 2015-07-28   |   分类于 MySQL ， 数据库

前面已经讲到过 MySQL数据类型选择 的原则，前提是我们需要了解MySQL数据库的数据类型，以及他们之间的区别和效率，那么这篇文章就来说明MySQL数据类型。

  • 整数类型
    整数类型包括TINYINT(8 bit)、SMALLINT(16 bit)、MEDIUMINT(24 bit)、INT(32 bit)、BIGINT(64 bit)。
    整数类型的存储范围为(-2)^(N-1) ~ 2^(N-1)，N表示存储的位数。同时整数类型有UNSIGNED属性，表示不允许负数，这使得存储范围的上限提高一倍。
    在存储上整数类型都是定长的，我们平时建表时定义的INT(11)表示数字显示长度而非存储长度，对于计算和存储来说INT(1)和INT(11)是一回事。

  • 实数类型
    实数类型包括FLOAT(32 bit)、DOUBLE(64 bit)、DECIMAL。
    这些都是存储精确的小数数据。DECIMAL最多允许存储65个数字，它只是一种存储格式，由MySQL的服务器自身实现其高精度计算，在计算中DECIMAL会转换成DOUBLE类型计算。
    实数运算是需要额外的空间和计算开销的，相比整数类型，我们应该优先考虑整数类型存储，只在必要使用实数存储的情况下考虑实数类型。在实际应用中，我们也可以通过BIGINT类型替换DECIMAL来存储，只需要将数据乘以相应的倍数即可。

  • 字符串类型
    字符串类型包括VARCHAR(65535)，CHAR。
    VARCHAR数据类型存储可变长字符串，额外使用1-2个字节来存储字符串长度，在ROW_FORMAT!=FIXED的情况下，它比CHAR更加节省空间。而CHAR数据类型则是存储定长字符串。
    逻辑上，VARCHAR和CHAR是相同的，存储上，因为CHAR是定长，所以MySQL才采用空格填充的方式以方便比较。在存储和检索时，VARCHAR数据类型会保留字符串末尾的空格符，而CHAR数据类型则会删除末尾的空格符。
    综上所述，以下情况比较适合采用VARCHAR数据类型：
    1、 列数据的长度不均匀，方差偏大。
    2、 列更新少。如果更新长度大于原有长度，则MySQL需要做很多多余工作来扩展空间存储新数据。
    3、 数据字符集复杂。如UTF-8，每个字符采用不同的字节数。
    以下情况适合采用CHAR数据类型：
    1、 经常变更的数据。定长的CHAR类型不容易产生碎片。
    2、 数据非常短的列或者长度变化不大的列。

  • 文本类型
    文本类型包括TINYBLOB、SMALLBLOB、BLOB、MEDIUMBLOB、LONGBLOB以及TINYTEXT、SMALLTEXT、TEXT、MEDIUMTEXT、LONGTEXT。
    它们之间仅有的不同就是BLOB类型是二进制类型，没有排序规则和字符集，TEXT是字符类型，有排序规则和字符集。
    对于文本类型，MySQL只针对文本的一部分进行排序和索引，排序是按照max_sort_length字节来排序的。

  • 枚举类型
    ENUM
    对于MySQL内部将值在列表中的位置保存为整数，并且在表的.frm文件中保存一份映射关系，所以枚举实际存储的是整数，在查找的时候会转换为字符串，这个你可以验证一下，同时在排序的时候也是按照这个位置来排序的，可以同field()函数改变排序规则。

  • 日期和时间类型
    日期和时间类型包括：TIMESTAMP(32 bit)、DATETIME(64 bit)。
    MySQL存储时间的最小精度是秒。DATETIME的时间范围是1001年至9999年，TIMESTAMP的时间范围为1970年到2038年。DATETIME的存储格式为整形，以文本的方式存储在数据库中，TIMESTAMP的存储格式则在各个版本中不同，它的存储值与时区相关，默认为NOT NULL，默认值为当前时区的当前时间。

MySQL的数据类型有很多，一些不常用的类型就不做介绍了，每一种类型都有它自己的特性，在一定的场景中都能表现出超越其他类型的优势，所以，我们需要认真思考需求和实际所需，参考前面所提的数据类型选择原则，根据项目中数据的变化来决定采用哪种数据类型存储。

MySQL数据类型选择

发表于 2015-07-26   |   分类于 MySQL ， 数据库

因为MySQL支持的数据类型很多，在数据库建表时，选择方案也就很多，每种都可以作为需求列的存储类型，但是确不是最合适的，选择正确的、合适的数据类型对于获得高性能是至关重要的。在做数据类型选择时有以下几条原则：

  • 大的不一定是最好的
    服务器总是繁忙的，能省事点就应该省事点。在数据类型的数据范围内能够满足需求不会溢出的前提下，我们应该优先考虑最小数据类型。因为小数据类型相比大数据类型，它们占用更少的磁盘、内存和CPU缓存空间，并且处理小数据类型的数据是占用CPU周期也更少。这也正是我们优化数据库所想要看到的结果，既然可以达到这样的效果，为什么不用呢？

  • 大简至美，大美至简
    复杂的事物对于我们来说，思考的就要更多，脑细胞就死的更快。这对于计算机是同理的，复杂的数据类型计算量会比简单的数据类型要大。所以简单的数据类型相比复杂的数据类型占用CPU周期更少。例如，整形运算要比字符类型操作简单。两个例子，一个是使用MySQL内建的类型存储时间日期，而不是字符串；另一个是用整形类型存储IP地址，PHP中通过ip2long()，long2ip()两个函数来对IP地址进行转换。

  • 丑陋的NULL
    在不是非常需要NULL值得情况下，在建表时尽量将列指定为NOT NULL。在MySQL中，存在NULL值的列存在以下缺点：
    ⒈ 如果查询中包含可为NULL的列时，对MySQL优化器来说是痛苦的，因为可为NULL列上的索引、索引统计和值都比较复杂，这前面一点说过了，能简单点就不要给自己使绊搞那么复杂。
    ⒉ 可为NULL值得列在存储数据时会占用更多的存储空间，而且在MySQL中还特么要特殊处理，这你能忍？
    ⒊ 当可为NULL的列被索引时，每个索引记录需要一个额外的字节，在MyISAM里甚至还可能导致固定大小的索引变成可变大小的索引。

不同的数据类型，存储的长度和范围不一样、允许的精度不一样，或者需要的物理空间不同，相同的大类型不同的小类型有时也有一些特殊的行为和属性，这些完全要看实际环境和需求来确定。所以选择合适的数据类型在数据库建立和优化中也是重要的一个环节。
