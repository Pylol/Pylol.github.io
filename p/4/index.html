SSL/TLS协议

　　SSL：安全套接字协议（Secure Sockets Layer）。它是客户机与服务器之间安全交换信息的协议，提供安全服务。它介于应用层与传输层之间。
　　注：TLS协议是SSL得升级版，在1999年，互联网标准化组织ISOC接替Netscape公司，发布TSL1.0版本。
　　　　　　

作用

　　之前记录的HTTPS协议就是基于SSL协议进行加密通信的。如果通信过程不进行信息加密，那么通信过程将存在以下几点风险：

  • 窃听风险：第三方可以获知通信内容
  • 篡改风险：如果第三方获取到通信信息，则可以篡改通信内容。
  • 冒充风险：第三方可以冒充他人身份进行通信。

　　由此可见，在互联网这种大而复杂的环境中，如果想要信息安全就必须解决这些通信问题。协议就是为了解决这些问题而设计的协议，它具备信息加密传播，身份证书，信息校验等功能。 　　

工作原理

　　SSL协议包含三种类型：握手协议（Handshake protocol），记录协议（Record protocol），警报协议（Alert protocol）。 　　

  • 握手协议
    　　握手协议记录客户端和服务端的信息，用于客户端与服务端之间相互验证和协议密钥。握手过程如下：

    　　　　　　　　　　　　　

1、客户端的问候
　　首先，客户端发起加密通信请求，如https请求等，这步叫做ClientHello请求。这一步中，客户端向服务器发送以下信息：
　　- 客户端可以支持的SSL版本号(VERSION)
　　- 一个客户端生成的随机数，用作生成密钥(RANDOM)
　　- 客户端可以支持的加密方法
　　　　密码套件格式：每个套件都以“SSL”开头，紧跟着的是密钥交换算法。用“With”这个词把密钥交换算法、加密算法、散列算法分开，例如：SSL_DHE_RSA_WITH_DES_CBC_SHA,表示把DHE_RSA(带有RSA数字签名的暂时Diffie-HellMan)定义为密钥交换算法；把DES_CBC定义为加密算法；把SHA定义为散列算法。
　　- 客户端可以支持的压缩方法。
　　- 会话ID。
　　- 服务器名
　　　　2006年，TLS协议加入了一个Server Name Indication扩展，允许客户端向服务器提供它所请求的域名。在此之前是不允许发送服务器名的。

2、服务器的问候
　　服务器收到客户端的请求之后，做出回应，这个过程叫做ServerHello。这一步中，服务器向客户端传送如下信息：
　　- 一个SSL协议版本。比较客户端支持的最高版本和服务器支持的最高版本，取较低的一个。
　　- 一个随机数。用于密钥的生成。
　　- 从客户端发送过来所支持的加密算法中选取一个返回给客户端。
　　- 从客户端发送过来所支持的压缩方法中选取一个返回给客户端。
　　- 会话ID。

　　这样一个来回，客户端和服务端相互都知道的内容：SSL版本号；加密算法；压缩方法；有关密钥的两个随机数。这些交换完后，双方开始身份鉴别和密钥的交换。

3、服务器亮出身份，问候完毕
　　这一步骤中，服务器发送给客户端的信息包括：
　　- 证书：证书中包含公钥，颁发机构等信息。
　　- 密钥交换：这个视加密算法而定。
　　- 证书请求：如果服务器需要验证客户端的身份，则会发送客户端证书验证的请求。
　　- 握手完毕。

上图中可见，服务器向客户端发送了三家机构颁发的证书。证书中包含了服务器端的公钥(subjectPublicKey)以及公钥的哈希值(SHA, shaWithRSAEncryption)，这个地方应该是验证公钥的有效性。最后服务器告诉客户端问候结束(Server Hello Done)。因为没有证书请求，所以没有这部分信息。

4、客户端验证服务端身份， 并亮出自己的身份
　　客户端收到服务端的信息后，并不是马上做出回应，而是先验证服务器端的身份（证书）。如果证书不是可信任机构或组织颁发，或者证书中的信息与实际不一致，或者证书已过有效期等不可靠信息，则会向访问者显示一个警报，让用户选择是否继续。如果证书验证没有问题则取出证书中的公钥并回应服务器：
　　- 一个随机数。
　　- 编码改变通知。表示随后的信息都将用双方商定好的加密方法和密钥进行加密。
　　- 客户端握手结束通知。这一项同时也是前面发送的所有内容的hash值，用来供服务器校验。
　
　　Encrypted PreMaster Secret即第三个随机数，成为pre-master-key。上面图中Change Cipher Spec是告知服务器后面的信息要用商议好的加密方式来加密传输；Encrypted Handshake Message就是把之前发送的所有内容进行hash发给服务器进行校验。
　　
　　第三个随机数发送给服务器，然后按照PRF算法计算得到主密钥。（生成主密钥的算法SSL和TLS略有不同。）(具体不清楚)

5、服务器最后的回应
　　服务器收到客户端第三个随机数pre-master-key后，计算生成本次会话所用的会话“会话密钥”，然后向客户端发送如下信息：
　　- 编码改变通知。
　　- 服务器握手结束。这一项同时也是前面发送的所有内容的hash值，用来供客户端校验。
　
　　上面信息的意思跟客户端回复服务端是一样的意思。

PHP多线程

　　PHP本身并不支持多线程，但是PHP可以通过扩展引用来实现多线。当然，有许多高手大能们通过各种方式去模拟多线程工作的方式来实现，比如：fsockopen，curl_multi，popen，pcntl_fork，proc_open等方式。这些我都不知道是什么，只有回头再去研究研究。
　　
　　这篇笔记只记录两种方式来实现PHP的多线程，一种通过并发模拟，一种通过扩展，即curl_multi，pthreads。

　　CURLOPT_URL：表示请求的链接地址。
　　CURLOPT_RETURNTRANSFER：表示是否需要返回。
　　CURLOPT_TIMEOUT：表示请求执行的最长时间。
　　CURLOPT_USERAGENT：表示请求的客户端。
　　最后执行请求回去返回结果，完成请求。

一、curl_multi模拟并发请求

　　
　　手册上curl_multi函数系列介绍的非常少，之前没接触过这块的，一些地方还是比较难理解。所以先记录解释一下一些函数的含义：

  • curl_multi_init( void )
    　　初始化一个curl_multi句柄，然后将多个curl句柄传递给此句柄。

  • curl_multi_add_handle(resouce $mh, resource $ch)
    　　向curl_multi()句柄传递curl句柄。第一个参数为curl_multi句柄，第二个参数为curl句柄。

  • curl_multi_exec(resource $mh, int $flag)
    　　发起curl_multi请求。第一个参数为curl_multi句柄，第二个用来记录未完成请求数。返回值为定义于cURL预定义常量中的cURL代码

  • curl_multi_info_read(resouce $mh [,int $msg_in_queue=NULL])
    　　获取当前解析的cURL信息。第一个参数为curl_multi句柄， 第二个参数为仍在消息队列中的请求数。成功时返回相关信息的数组，失败返回FALSE。

  • curl_multi_getcontent($ch)
    　　如果设置了CURLOPT_RETURNTRANSFER,则返回获取的输出文本流。第一个参数为curl句柄。

  • curl_multi_remove_handle($mh, $ch)
    　　从curl_multi句柄中移除curl句柄，若移除的curl句柄正在执行，则会停止处理过程中所有的传输任务。

  • curl_multi_close($mh)
    　　关闭一组cURL句柄。

执行步骤：
1.通过 curl_multi_init()初始化curl_multi句柄。
2.通过curl_init()获得curl句柄，并通过curl_multi_add_handle()添加至curl_multi句柄中
3.持续调用curl_multi_exec()
4.通过curl_multi_getcontent()获取运行结果
5.调用curl_multi_remove_handle()移除curl句柄并通过curl_close()注销句柄
6.调用curl_multi_close();

pthreads的php扩展

　　POSIX线程，简称pthreads，是线程的POSIX标准，该标准定义了创建和操纵线程的一整套API。
　　通过使用pthreads扩展，这个才是真正意义上多线程，程序运行时会再php进程下创建多个线程并发执行代码片段，从而达到性能的高效。步骤同普通curl请求，就不多记录。

phreads扩展的安装
　　1.下载pthreads扩展包
　　2.将php_pthreads.dll文件复制到/path/to/php/ext/目录下。
　　3.将pthreadVC2.dll文件复制到/path/to/php/目录下。(有些安装教程里面说要把这个文件也复制到C://windows/system32目录下，但是我没有复制也能正常运行，如果不行的话，复制过去吧)。
　　4.在php.ini配置文件中加入extension=php_pthreads.dll扩展信息。
　　5.将pthreadVC2.dll的路径加入到系统的环境变量PATH中。
